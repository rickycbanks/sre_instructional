# Module 5: Incident leadership is part technical and part communication

A lot of SRE practice is about staying structured when systems and people are both under stress. That means you need to understand not only **what** to do during an incident, but **how** to coordinate the work around it.

## The incident lifecycle

1. **Detection** — a signal suggests user pain or elevated risk.
2. **Triage** — confirm impact and scope.
3. **Mitigation** — stop the bleeding.
4. **Resolution** — restore stable service.
5. **Learning** — capture the conditions that allowed the incident.

The key distinction is between mitigation and root cause. In strong operational practice, people say things like:

> "I would first restore the user path, even if I do not fully understand the root cause yet."

## Roles reduce chaos

Clear roles matter because they preserve decision quality:

- **Incident commander:** keeps the big picture, prioritizes, delegates
- **Operations lead:** executes the technical actions
- **Communications lead:** keeps stakeholders informed with accurate, calm updates

One common trap is letting the incident commander also become the keyboard operator. That sounds efficient, but it usually degrades coordination.

## Blameless does not mean accountability-free

A blameless postmortem still has accountability:

- document what happened
- identify the enabling system conditions
- assign follow-up actions with owners
- improve controls so the same class of failure is harder to repeat

That is stronger than simply saying, "We should be more careful."

## Interview rubric: "How would you talk to leadership after the outage?"

Strong answers usually:

- summarize impact clearly
- avoid scapegoating
- explain what was done to restore service
- name the follow-up system changes
