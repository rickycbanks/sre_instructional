# Module 11: Kubernetes is an operations platform, not an excuse to stop thinking

Kubernetes often exposes broader operational judgment questions:

- can you stop a bad rollout?
- do you understand what health probes are telling you?
- can you separate capacity problems from release problems?
- do you know how to make the next deployment safer?

## The control points worth knowing cold

- **readiness probes** protect live traffic from unready pods
- **liveness probes** recover stuck processes, but can also create churn if misused
- **requests and limits** shape scheduling and resource isolation
- **HPA** responds to demand, but only if the scaling signal matches reality
- **rollout pause/rollback** is how you stop blast radius during a bad deploy

## Beyond rollouts: how Kubernetes workload organization affects reliability

The higher-leverage explanations go beyond "use probes and rollback." They often mention:

- **Deployments** for stateless workloads where replicas are interchangeable
- **StatefulSets** when identity, ordered startup, or persistent storage semantics matter
- **DaemonSets** when every node needs a copy of the workload, such as an agent or collector
- **Pod Disruption Budgets** to protect availability during maintenance
- **affinity / anti-affinity** to spread or group workloads intentionally

These choices shape whether upgrades, node drains, and failures behave predictably.

## Scheduling and capacity questions often hide behind Kubernetes questions

If a cluster looks healthy but pods keep getting evicted or throttled, a strong explanation usually considers:

- whether requests are realistic
- whether limits are too aggressive
- whether autoscaling is driven by the right metric
- whether the workload is placed across failure domains appropriately

That is Kubernetes reasoning, but it is also classic SRE reasoning: understand the control loops and protect the user path.

## Strong rollout answers

When a deployment starts hurting users, a strong candidate usually says some version of:

1. check whether the issue aligns with the new rollout
2. stop, pause, or roll back if the signal is strong enough
3. preserve evidence
4. restore the user path
5. improve the release guardrails before the next deploy
