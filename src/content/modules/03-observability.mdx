# Module 3: Observability that helps you decide what to do next

Good observability is not a wall of charts. It is a system that helps an engineer answer three questions fast:

1. **Are users affected?**
2. **Where in the request path is the problem surfacing?**
3. **What is the safest next action?**

## The golden signals still matter

- **Latency:** how long work takes
- **Traffic:** how much demand exists
- **Errors:** explicit or implicit failure
- **Saturation:** how close the system is to its limits

The reason interviewers ask about them is not because they want a memorized list. They want to see that you know what each signal is useful for during investigation.

## Metrics, logs, and traces each answer different questions

### Metrics

Best for:

- fast detection
- trend comparisons
- SLO and alerting logic

### Logs

Best for:

- precise event evidence
- understanding failure context
- seeing unexpected branches, retries, and bad inputs

### Traces

Best for:

- request-path timing
- dependency breakdown
- understanding whether the slow part moved after a change

## A reliable triage pattern

When something breaks, a strong flow is often:

1. confirm user impact
2. compare against recent changes
3. isolate the failing layer
4. mitigate the most likely blast-radius source
5. investigate root cause after stability returns

That sequence sounds obvious, but in interviews many candidates skip directly from symptom to fix.

## Interview shortcut: "What would you check first?"

Your answer should usually contain some version of:

- start with user-facing SLI impact
- compare with the last known good state or recent deploys
- use metrics to narrow, traces/logs to explain
- keep mitigation separate from deep root-cause work
