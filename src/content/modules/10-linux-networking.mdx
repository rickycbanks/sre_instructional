# Module 10: The debugging fundamentals strong operators still rely on directly

Even in cloud-native environments, Linux and networking basics still matter because these are the skills that help you debug without magical thinking.

## Linux: start with the bottleneck you suspect

Common first-principles questions include:

- is the host CPU-bound, memory-bound, or I/O-bound?
- is a process healthy, restarting, blocked, or leaking?
- did the kernel start swapping?
- are limits or file descriptors being exhausted?

Strong answers choose commands that validate the suspected bottleneck, not commands that are merely familiar.

## Networking: isolate the layer

A clean troubleshooting flow separates:

- **DNS** — can the name resolve?
- **TCP** — can a connection be established?
- **TLS** — can the handshake succeed?
- **HTTP** — is the application responding correctly?

It is easy to blur these together. The strongest explanations test the narrowest layer first.

## Strong-answer habits

- start with the symptom you have, not the fix you want
- choose the command that narrows the problem most directly
- say what you expect to learn from running it
- explain what your next branch would be depending on the output
