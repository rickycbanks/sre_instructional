# Module 12: Containers and workloads that are built for production, not just for demos

Many technically strong candidates know how to build a container that works. Fewer can explain why one image is safer, faster, and easier to operate than another. That gap shows up quickly in interviews.

## What good container organization looks like

A production-friendly image usually aims for:

- **small runtime footprint**
- **predictable startup behavior**
- **non-root execution**
- **clear process model**
- **minimal dependency sprawl**

The practical tools behind that are not mysterious:

- multi-stage builds
- tight `.dockerignore`
- pinned and intentionally updated base images
- keeping build tools out of the final runtime image
- setting a clear `ENTRYPOINT` and sane default arguments

## Image optimization is also operational optimization

Slimmer images do more than save storage:

- they pull faster during scale-out and node replacement
- they reduce the number of packages you have to patch or scan
- they make cold starts and rollout blast radius smaller

That is why image quality belongs in an SRE conversation, not just a developer productivity one.

## Kubernetes workload organization

Once the image is good, the next question is how to run it.

### Use a Deployment when:

- replicas are interchangeable
- identity does not matter
- rolling updates are the default control model

### Use a StatefulSet when:

- stable network identity matters
- storage follows the replica
- ordered startup or shutdown matters

### Use an init container when:

- setup work must finish before the app starts
- you want to keep bootstrap tooling out of the main container

### Use a sidecar when:

- the helper lifecycle belongs with the workload
- the operational value outweighs the extra resource and debugging complexity

## Advanced Kubernetes controls worth being able to discuss

- **requests and limits** as scheduling and isolation signals
- **Pod Disruption Budgets** to preserve availability during maintenance
- **node affinity / anti-affinity** to shape placement and failure domains
- **imagePullPolicy** and immutable tags for repeatable rollouts
- **evictions** as a symptom of resource pressure or poor sizing

## Strong interview framing

If asked how you would improve a container platform, a strong answer often includes:

1. improve the artifact first: build size, hardening, reproducibility
2. choose the right workload controller for the application’s behavior
3. set sane requests, limits, probes, and disruption controls
4. connect everything back to startup speed, blast radius, and operator confidence
