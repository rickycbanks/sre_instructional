# Module 7: Service meshes are traffic policy engines, not magic

A service mesh such as Istio or Linkerd can standardize:

- service-to-service encryption
- retries and timeouts
- traffic splitting
- observability hooks
- policy enforcement

That can be powerful, but only if the team understands the tradeoffs.

## Where meshes help

- multi-service environments with repeated traffic policy needs
- consistent mTLS between workloads
- progressive delivery patterns like canaries
- centralized visibility for east-west traffic

## Where meshes can hurt

- added control-plane complexity
- harder debugging when engineers do not understand proxy behavior
- retry storms if timeouts and backoff are configured poorly

Strong interview answers usually sound balanced:

> "A mesh can reduce repeated application logic and standardize policy, but it also adds a proxy and control-plane layer that the team must be able to operate confidently."

## Reliability patterns you should be able to discuss

- retries should be bounded and paired with timeouts
- circuit breaking is valuable when dependencies degrade
- traffic shifting is only safe when backed by observability and rollback criteria
- mTLS protects service-to-service communication, but certificate lifecycle still needs operational care
